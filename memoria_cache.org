# ============================================================
#  DIAPOSITIVA 6 — PRINCIPIO DE LOCALIDAD (DEFINICIÓN)
# ============================================================
* Principio de Localidad de Referencia
:PROPERTIES:
:reveal_background: #0a0f1e
:END:

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div class="concepto-box" style="margin-bottom:16px;">
  <strong>Definición (Stallings, p. 135):</strong><br/>
  Durante la ejecución de un programa, las referencias a memoria
  por el procesador — tanto instrucciones como datos — tienden
  a <em>agruparse</em>. No todas las posiciones de memoria son
  igualmente probables de ser accedidas en un momento dado.
</div>
#+END_EXPORT

#+BEGIN_EXPORT html
<div class="col2">
<div>
#+END_EXPORT

#+ATTR_REVEAL: :frag (appear appear appear)
*¿Por qué ocurre?*
- Bucles: mismas instrucciones una y otra vez
- Subrutinas: se invocan repetidamente
- Estructuras de datos: arrays accedidos secuencialmente

#+BEGIN_EXPORT html
</div>
<div>
#+END_EXPORT

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div style="font-size:0.62em; background:#0d1e30;
            border-radius:8px; padding:14px; text-align:left;">
<p><strong style="color:#f0c040;">Denning (1972)</strong> formuló
   3 afirmaciones:</p>
<ol style="line-height:1.7;">
  <li>Las referencias son <strong>no uniformes</strong></li>
  <li>La distribución cambia <strong>lentamente</strong></li>
  <li>Pasado reciente ≈ Futuro inmediato
      <span style="color:#a8e6cf;">(alta correlación)</span></li>
</ol>
</div>
#+END_EXPORT

#+BEGIN_EXPORT html
</div>
</div>
#+END_EXPORT

#+BEGIN_NOTES
INTEGRANTE 1 — ~2 min
Explicar que este principio es el FUNDAMENTO de toda la jerarquía.
Sin localidad de referencia, la caché no serviría de nada.
Vincular con el juego anterior: "Bob tendía a usar las mismas
carpetas (temporal) y buscar las vecinas (espacial). Eso es
localidad. Y los programas hacen exactamente lo mismo."
Mencionar que Denning (1972) fue quien lo formalizó matemáticamente.
#+END_NOTES

# ============================================================
#  DIAPOSITIVA 7 — LOCALIDAD TEMPORAL
# ============================================================
* Localidad Temporal
:PROPERTIES:
:reveal_background: #0a0f1e
:END:

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div class="concepto-box temporal">
  <strong>Localidad Temporal</strong> (Stallings p. 136):<br/>
  Si una posición de memoria <code>x</code> fue referenciada en el
  tiempo <code>t</code>, es <em>probable</em> que vuelva a ser
  referenciada en un tiempo cercano a <code>t</code>.
</div>
#+END_EXPORT

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div style="font-size:0.68em; background:#0d1e30; border-radius:8px;
            padding:12px 18px; margin:8px 0; text-align:left;">
<strong style="color:#f0c040;">Ejemplo en código:</strong>
<pre style="margin:6px 0; color:#a8e6cf; font-size:1em;">
for (i = 0; i &lt; 1000; i++) {
    suma += array[i];   ← variable "suma" se usa 1000 veces
}
</pre>
<p style="margin:4px 0;">
  La variable <code>suma</code> y el contador <code>i</code> son
  accedidos repetidamente → <strong>alta localidad temporal</strong>.
</p>
</div>
#+END_EXPORT

#+ATTR_REVEAL: :frag (appear appear)
*Cómo la explota la caché:*
- Mantiene instrucciones y datos *recientemente usados* en caché
- Organiza jerarquía de múltiples niveles (L1 → L2 → L3)

#+BEGIN_NOTES
INTEGRANTE 1 — ~1.5 min
El ejemplo del bucle es perfecto para la clase. Preguntar:
"En este for, ¿cuántas veces se lee 'suma'? ¡1000 veces!
¿Tendría sentido ir a la RAM 1000 veces para buscarla?
Con caché, la buscamos UNA vez, la traemos y la tenemos
disponible para las 999 referencias siguientes."
Dibujar en el aire la idea de "mantener cerca lo que ya usaste".
#+END_NOTES

# ============================================================
#  DIAPOSITIVA 8 — LOCALIDAD ESPACIAL
# ============================================================
* Localidad Espacial
:PROPERTIES:
:reveal_background: #0a0f1e
:END:

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div class="concepto-box espacial">
  <strong>Localidad Espacial</strong> (Stallings p. 136):<br/>
  Si la posición <code>x</code> es referenciada en el tiempo
  <code>t</code>, es probable que las posiciones
  <code>x−k</code> a <code>x+k</code> sean referenciadas en
  un futuro cercano (para un <code>k</code> relativamente pequeño).
</div>
#+END_EXPORT

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div style="font-size:0.68em; background:#0d1e30; border-radius:8px;
            padding:12px 18px; margin:8px 0; text-align:left;">
<strong style="color:#f0c040;">Ejemplo en código:</strong>
<pre style="margin:6px 0; color:#a8e6cf; font-size:1em;">
for (i = 0; i &lt; N; i++) {
    resultado[i] = A[i] + B[i];   ← acceso secuencial a arrays
}
</pre>
<p style="margin:4px 0;">
  Si accedo a <code>A[0]</code>, lo probable es que pronto
  acceda a <code>A[1]</code>, <code>A[2]</code>...
  → La caché trae un <em>bloque</em> completo de memoria.
</p>
</div>
#+END_EXPORT

#+ATTR_REVEAL: :frag (appear appear)
*Cómo la explota la caché:*
- Transfiere *bloques* completos (no palabras sueltas)
- Usa mecanismos de *prefetching* (precarga anticipada)

#+BEGIN_NOTES
INTEGRANTE 1 — ~1.5 min
Analogía del escritorio: "Cuando Bob va al archivero a buscar la
carpeta del Proyecto A, ¿qué hace? Trae también las carpetas
vecinas: presupuesto, cronograma, contactos. Eso es exactamente
lo que hace la caché: trae un BLOQUE de memoria, no solo el
byte que pediste."
Mencionar que el tamaño del bloque (línea de caché) típicamente
es de 64 bytes en procesadores modernos.
CIERRE FASE 1: "Con esto ya tienen la intuición. [Nombre 2] va
a construir ahora la estructura formal."
#+END_NOTES

# ============================================================
#  DIAPOSITIVA 12 — TIEMPO DE ACCESO VS CICLO
# ============================================================
* Rendimiento: Acceso vs. Ciclo vs. Tasa de Transferencia
:PROPERTIES:
:reveal_background: #0a0f1e
:END:

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div class="col2">
<div>
#+END_EXPORT

#+ATTR_REVEAL: :frag (appear appear appear)
- 📏 *Tiempo de Acceso* ($T_A$): tiempo desde que se presenta la dirección hasta que el dato está disponible
- 🔄 *Tiempo de Ciclo* ($T_C$): $T_A$ + tiempo extra de recuperación. Es el mínimo tiempo entre dos accesos consecutivos ($T_C \geq T_A$)
- 📡 *Tasa de Transferencia* ($R$): para RAM $= 1/T_C$ ; para no-RAM: $T_n = T_A + n/R$

#+BEGIN_EXPORT html
</div>
<div>
#+END_EXPORT

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div style="background:#0d1e30; border-radius:8px; padding:14px;
            font-size:0.62em; text-align:left;">
<p><strong style="color:#f0c040;">¿Por qué T_C &gt; T_A?</strong></p>
<p>En memorias DRAM, la lectura es <em>destructiva</em>: leer un dato
   lo borra. El sistema debe <strong>reescribirlo</strong> antes de
   permitir otro acceso.</p>
<p style="margin-top:8px;">
  <strong>Fórmula para no-RAM (Stallings Ec. 4.1):</strong>
</p>
<pre style="color:#a8e6cf; margin:4px 0;">
  T_n = T_A + n/R
  donde n = bits a transferir
</pre>
</div>
#+END_EXPORT

#+BEGIN_EXPORT html
</div>
</div>
#+END_EXPORT

#+BEGIN_NOTES
INTEGRANTE 2 — ~1.5 min
Dar el ejemplo concreto: "Si la RAM tiene T_A = 60 ns y necesita
10 ns extra para regenerar, T_C = 70 ns. Aunque el dato llegó en
60 ns, no puedes hacer otro acceso hasta que pasen 70 ns."
La fórmula de tasa de transferencia para no-RAM es importante:
"Si transferimos n=128 bits, T_A = 20 ms, R = 2 Mbps en una cinta:
T_128 = 20ms + 128/(2×10^6) = 20ms + 0.064ms"

#+BEGIN_NOTES
INTEGRANTE 4 — ~2 min
Explicar cada algoritmo brevemente:
LRU: "Razona: si no lo usé recientemente, probablemente no lo
necesite. Para 2-way es trivial: 1 bit por línea."
FIFO: "El primero en entrar es el primero en salir. Como una cola.
La anomalía de Bélády es un fenómeno curioso: más caché puede dar
PEOR hit ratio con FIFO (no con LRU)."
LFU: "Cuenta referencias. Problema: una línea muy usada al inicio
nunca se expulsa aunque ya no la uses."
RAND: "¡Funciona sorprendentemente bien! Smith (1982) mostró que
es solo ligeramente inferior a LRU. Y el hardware es trivial."
Pregunta al público: "¿Cuál usarían para la L1 del CPU si
el hardware debe ser ultrarrápido?" → LRU aproximado o RAND.

  <div style="color:#2980b9; font-weight:700; font-size:1.3em; margin-bottom:6px;">

  <p style="color:#d63031;">❌ Posible cuello de botella</p>
  <p style="color:#aaa; font-size:0.9em; margin-top:6px;">
    Usado en: IBM z13 L1 y L2 (Stallings p. 190)
  </p>

            font-size:0.62em; background:#0d1e30;">
  <div style="color:#e67e22; font-weight:700; font-size:1.3em; margin-bottom:6px;">
    💾 Write-Back (Copy-Back)
  </div>
  <p>Escribe <strong>solo en caché</strong>. Marca línea con
     <em>Dirty Bit</em>. Escribe a RAM solo al expulsar la línea.</p>
  <p style="color:#00b894;">✅ Mínimo tráfico de bus</p>
  <p style="color:#00b894;">✅ Mejor rendimiento general</p>
  <p style="color:#d63031;">❌ RAM puede estar desactualizada</p>
  <p style="color:#d63031;">❌ Coherencia compleja</p>
  <p style="color:#aaa; font-size:0.9em; margin-top:6px;">
    Usado en: IBM z13 L3 y L4 (Stallings p. 191)
  </p>
</div>

#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div class="concepto-box" style="font-size:0.62em; margin-top:6px;">
  <strong>Ejemplo 5.2 (Stallings p. 182):</strong>
  Línea de 32 bytes, RAM tarda 30 ns/palabra (4B).
  Write-Back escribe la línea 1 vez al expulsar (8×30 = 240 ns).
  Write-Through escribe cada vez que modifica. Si se modifica
  <strong>&gt; 8 veces</strong> antes de expulsar → Write-Back gana.
</div>
#+END_EXPORT

#+BEGIN_NOTES
INTEGRANTE 4 — ~2.5 min
Usar la analogía del cuaderno de borrador (caché) vs. el libro
oficial (RAM):
"Write-Through: cada vez que escribes algo en tu borrador,
inmediatamente copias exactamente lo mismo al libro oficial.
El libro SIEMPRE está al día."
"Write-Back: escribes en el borrador con un post-it de 'sucio'.
Solo pasas al libro oficial cuando necesitas ese espacio del borrador."
El ejemplo numérico: "Si la línea se escribe solo 1 vez antes de
expulsarse, Write-Back = 240 ns, Write-Through = 30 ns → WT gana.
Si se escribe 9 veces: WB = 240 ns, WT = 270 ns → WB gana."
Mencionar IBM z13: L1 y L2 usan WT (porque tienen alta velocidad
hacia L3), L3 y L4 usan WB (tráfico hacia RAM más costoso).
#+END_NOTES

*Número y Tipo de Cachés*
#+ATTR_REVEAL: :frag appear
#+BEGIN_EXPORT html
<div class="concepto-box" style="font-size:0.65em;">
  Stallings p. 184 — <strong>Caché Unificada</strong> vs.

- *Dividida* (Harvard-style): caché de instrucciones (I-cache) + datos (D-cache). Evita contención con pipelining → *estándar en L1 moderno*

#+BEGIN_NOTES
INTEGRANTE 4 — ~2 min
Tamaño de línea: "Si la línea es muy pequeña, el principio de
localidad espacial no se aprovecha. Si es muy grande, trae datos
que nunca usarás y cada miss tarda más. 64 bytes es el estándar
en Intel y AMD modernos."
Split cache: "La razón es el pipeline. Cuando el CPU está ejecutando
instrucción N y al mismo tiempo precarga instrucción N+1, necesita
acceder a la I-cache. Si hay una sola caché y la instrucción N
también lee datos en ese ciclo, hay CONTENCIÓN. Con cachés separadas,
instrucciones y datos no compiten."

Revelar la tabla fila por fila, explicando cada decisión:
Ref 1: "Fallo — caché vacía, cargamos 1. Cola FIFO: [1]"
Ref 2: "Fallo — aún hay espacio, cargamos 2. Cola: [1,2]"
Ref 3: "Fallo — último espacio, cargamos 3. Cola: [1,2,3]"
Ref 1: "ACIERTO — 1 ya está en caché. Cola sin cambio: [1,2,3]"
Ref 4: "Fallo — caché llena. FIFO expulsa al PRIMERO EN ENTRAR:
  el bloque 1 (entró primero). Cola: [2,3,4]. ¡Pero ahora si
  necesitamos el 1 de nuevo, habrá fallo!"
Ref 1: "Fallo — 1 fue expulsado. Cargamos 1, expulsamos 2."
Ref 2: "Fallo — 2 fue expulsado. Cargamos 2, expulsamos 3."
Resaltar la paradoja: "El bloque 1 que acabábamos de usar en el
paso 4 fue expulsado en el paso 5. LRU habría expulsado a 3
(el menos recientemente usado) y el paso 6 habría sido acierto."

#+ATTR_REVEAL: :frag (appear appear appear appear appear)
1. 🎯 El *Principio de Localidad* (temporal y espacial) hace posible toda la jerarquía de memoria
2. 🏗️ La jerarquía balancea *velocidad, costo y capacidad* — sin ella el CPU esperaría casi todo el tiempo
3. ⚡ La caché es transparente al programador pero crítica para el rendimiento — su diseño determina el *hit ratio*
4. 🔀 *Asociativo por conjuntos* es el mejor compromiso: casi tan flexible como el asociativo puro, casi tan simple como el directo
5. 📝 *Write-back* minimiza tráfico de bus; *Write-through* maximiza coherencia. Los diseñadores modernos usan ambos en distintos niveles
